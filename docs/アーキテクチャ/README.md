# architecture — 構造の authority（定義）

ここは Koma の **構造（振る舞い以前）の定義**を置く場所。人間が著者で、**構造判断はここに固定する**。

Koma は spec-runner（request.md → PR のパイプライン）で実装する。実装は `packages/` / `apps/` を書き換えるが、
**何をどこに置き、どの依存を許すか**という構造の定規はこのディレクトリが持つ。実装はこれを follow し、構造を変えたいときは人間がここを編集する（＋理由を ADR）。

## このディレクトリが書くもの / 書かないもの

| | 内容 | authority |
|---|---|---|
| **書く（構造）** | 層の割り当て / 依存方向（DSM の edge）/ port・seam の境界 / 集約・値オブジェクトの境界 / 不変条件 / 型の所在 / 状態集合・許可遷移 | この `docs/アーキテクチャ/` |
| **書かない（振る舞い）** | メソッド・ユースケースが「何をするか」＝ アルゴリズム / 空き枠計算の手順 / 解決順 / routing / 業種別の product 選択 / 実装の段階計画 | spec（`specrunner/changes/`）・request |

> **litmus**: 「層・依存・境界・集約・不変条件・状態集合の話か？」→ YES なら構造（ここ）。「関数/ユースケースが何をするかの話か？」→ YES なら spec 側。迷ったら architecture には書かない。

振る舞いは「動的構造」としてのみ本ディレクトリに現れる（状態機械・実行時束縛・遷移不変条件＝`dynamic-model.md`）。それを駆動する手順は behavior であり、ここからは参照に留める。

## ファイル構成（断面＝View の階層）

| ファイル | 断面（View）| 内容 |
|---|---|---|
| **`model.md`** | 層・依存（Development）| 様式 / 層 / 許可された依存（DSM）/ 構造不変条件 B-x |
| **`domain-model.md`** | 型/データ（Logical / DDD）| 集約 / エンティティ / 値オブジェクト / 静的データの不変条件 / 拡張点 |
| **`dynamic-model.md`** | 動的構造（Logical / Dynamic）| 状態機械（Booking lifecycle）/ モジュール間束縛（イベント）/ 遷移不変条件 |

> **SoT 境界**: 正確な signature/型は **コードが正典**（各パッケージの `src/`）。本ディレクトリは陳腐化しない粒度（責務・契約の形・境界・不変条件）まで。
>
> **意図的非カバー**: コンポーネント単位の公開インターフェース（C4 Component）は、各パッケージが実在してから interface-first で `components.md` に起こす。コードの無い段階で signature を確定させない。

## 使い方

- **コード/振る舞いを書く** → `model.md`（どこに・何を import してよいか）＋ `domain-model.md`（どの型・どの集約）を読む。
- **構造に沿うか review** → 依存方向（DSM の edge）と B-x 不変条件、集約境界。
- **構造を変えたい** → 該当 doc を人間が編集し、理由を ADR に追記する。
